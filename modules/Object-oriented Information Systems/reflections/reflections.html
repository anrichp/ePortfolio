Unit 1 Reflection

Unit 1 was an excellent introduction to information systems; admittedly, I thought I understood what an information system was. Still, after researching an information system failure, I realised that I was lead astray by my definition of an information system.

Later I discovered that information systems are made up of five components hardware, software, data people, and process. Organisations often use these systems to collect, filter, process, create and distribute data.

An example of an information system could be Walmart’s retail management system which keeps track stock levels within the organisation and allows vendors and suppliers to access these stock levels to assign stock where necessary dynamically.

After understanding what information system is, we were tasked to participate in a discussion form that outlined an organisation and an information system failure. Unfortunately, I misunderstood what information systems were and used a medical treatment device controlled by a computer system as an example of an information system failure. The medical machine or device contains some of the five components of an information system; however, it does not help an organisation manage its resources and distribute information and data.

Going into Unit 2, I have a greater awareness of what an information system is, and this should allow me to better respond to my peers in the follow-up questions in the discussion forum.

Unit 2 Reflection

Unit 2 formed mostly a continuation of the discussion forum in Unit 1, where we further explored information systems’ failures in different contexts. Fascinatingly some of the largest organisations with ample resources are the ones who are failing miserably at producing information systems which meet the initial criteria.

This week, it has become clear that the organisations failed in crucial aspects of the development life cycle in most information system failures. In the two cases that I had explored in the discussion forums, it was clear that the failures resulted from insufficient planning, insufficient testing and a failure to analyse the organisation and its business needs to understand better how the IT assets co-exist with the organisation.

The more I read about these failures, the more I see the lack of processes and planning combined with politics and leaders’ opinions within the organisation that ultimately skew the information system’s outcome and success. We require a stringent system and process to be in place and good sign-off from organisations to prevent the system from wandering down a path of failure.

Unit 3 Reflection

Throughout this unit, I have gained some fundamental knowledge of object orientated programming using Python.

Object orientated programming is incredibly powerful as you can segment components within a system which allows you to work on a specific component without the fear of crippling an entire system.

I particularly find the Python __init__ method quite fascinating, the method allows you to pre-append a template of sorts to your program allowing the data structure of the class to be defined before you continue building the methods within that class. When creating a method within a class one has to be careful to include the “self” attribute which is incredibly important as the method is then able to access the state and variables of the class.

Looking forward to Unit 4, I am quite excited to work with more complex objects and understand inheritance and the role it plays in polymorphism.

Unit 4’s primary outcomes were to establish a greater understanding of objects, classes, class functions, class methods and inheritance.

I found Unit 4 quite overwhelming as I found the sheer volume of programming to be quite disorientating. I find that I have understood the most basic concepts of the desired outcomes however I find that certain concepts regarding inheritance are beyond my understanding at this point.

The purpose of Unit 5 was to extend the object-orientated design understanding from the previous Units in a way that allows us to design a piece of software visually and communicate this in a way that other programmers or software developers would understand.

I can see that UML is an integral part of the software engineering process. I found it quite challenging to decide how to layout the UML diagrams and which methods to use in which classes. Unit 9 Reflection

During Unit 9, we were introduced to the concepts of CRUD (Create, Read, Update and Delete) in MySQL databases, the codio exercises regarding MySQL was a great resource to understand the various queries used in CRUD. Additionally, we learned about NoSQL and the benefits of these databases, along with their best use cases.

This unit has made me reflect on the system implementation assignment and store the assignment’s data. After doing some research, I have come across ORM(Object Relational Mapping). ORM can be used to map classes to a database in a relational way. I have decided to use SQLAlchemy, which is an ORM for MySQL and Python.

Looking forward to Unit 10, I will start on my system implementation and work on mapping my classes to tables within the database. Additionally, I will need to do some additional reading on style guidelines for comments in my implementation.

I am reflecting on some of the decisions I made during the design of my system implementation project. I have found many shortcomings in my design. After researching a few alternatives, I am pretty excited by MongoDB as an alternative to traditional SQL databases, and I am, in retrospect, quite annoyed that I had not learned of MongoDB sooner. I believe using a NoSQL database would have been an exciting challenge for the system implementation project.

Looking forward to Unit 11, I will continue to work towards my system implementation project, which is due at the end of Unit 11. I will ensure that my classes in my code map correctly to my database, which will lay a good foundation for the code to follow.
